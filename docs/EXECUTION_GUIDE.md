## 🎯 DDD 理念核心原则

在审查阶段，我会重点关注以下原则是否被遵守：

### 分层职责
| 层级 | 职责 | 禁止事项 |
|------|------|----------|
| **core/** | 实体、接口定义 | 业务逻辑、I/O |
| **domain/** | 业务规则、策略 | I/O、async、第三方依赖 |
| **application/** | 用例编排、协调 | 直接数据库访问（需通过仓储） |
| **infrastructure/** | 外部适配器、实现 | 业务规则 |
| **web/** | API 路由、请求响应 | 业务逻辑 |

### 依赖规则
```
web → application → domain → core
                 ↘         ↗
              infrastructure
```
- 外层可以依赖内层
- 内层不能依赖外层
- 基础设施层依赖核心层的接口

## 📌 额外改进建议

除了本次重构的核心需求外，以下优化点值得关注：

### 短期优化
1. **React Query**：使用 React Query 替代自定义轮询，支持缓存和重试
2. **虚拟滚动**：信号列表超 100 条时启用虚拟滚动
3. **错误边界**：添加 ErrorBoundary 防止单点故障

### 中期优化
1. **WebSocket**：实时价格推送改用 WebSocket
2. **领域事件**：引入事件总线解耦信号产生与通知
3. **CQRS**：查询与命令分离，优化读性能

### 长期优化
1. **微服务拆分**：信号检测、持仓管理、通知服务独立部署
2. **消息队列**：使用 Redis Stream 处理实时 K 线
3. **多交易所**：抽象交易所接口，支持多交易所接入
