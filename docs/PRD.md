# CryptoRadar v1.0 - 产品需求与业务规则文档 (PRD)

## 1. 产品定位与核心红线
* **产品名称**：CryptoRadar (加密货币高胜率信号雷达)
* **核心目标**：7x24 小时监听市场，基于 `Pinbar + EMA60` 策略发现顺势波段交易机会。结合真实账户余额进行动态风控算仓，最后通过多渠道发送决策通知。
* **绝对红线 (Read-Only 约束)**：本系统为**纯监听与计算辅助工具**。系统内**严禁**包含任何向交易所发送真实交易指令（如下单、改单、平仓）的代码。API 仅限拉取行情与读取账户余额。

---

## 2. 核心业务流水线 (Core Workflow)
系统主循环为异步事件驱动，必须按以下顺序严格单向流转：
1. **行情接收**：通过 WebSocket 实时监听 K 线，仅处理已闭合 (is_closed=True) 的 K 线。
2. **配置热加载**：处理每根新 K 线前，从本地 SQLite 数据库异步读取最新的系统开关和风控参数。
3. **策略检测**：将闭合 K 线送入大脑，计算 EMA/ATR 指标，匹配 Pinbar 形态。
4. **只读余额获取**：若产生有效信号，调用交易所 API 读取当前真实的钱包余额和可用余额。
5. **风控算仓**：基于风控参数、当前余额、持仓笔数，推算建议的开仓杠杆倍数和具体数量。
6. **持久化入库**：将信号快照和算仓结果异步存入 SQLite 数据库。
7. **并发广播**：将决策信息渲染为 Markdown 格式，并发推送到 飞书、Telegram 等渠道。

## 3. 核心业务规则详细定义 (Business Rules)

### 3.1 策略规则引擎 (Pinbar + EMA60)
当且仅当满足以下**所有**条件时，才能生成有效信号 (Signal)：
* **趋势过滤**：
  * 做多 (LONG)：K线收盘价 > EMA60。
  * 做空 (SHORT)：K线收盘价 < EMA60。
* **形态识别 (Pinbar)**：
  * 实体比例：K线实体长度 (|Open - Close|) <= K线总长度 (High - Low) 的 25%。
  * 影线比例：有效长影线（多单看下影线，空单看上影线）必须 >= 实体长度的 2.5 倍。
* **波动率与防守过滤**：
  * 活跃度验证：K线总长度必须 > 1.2 * ATR(14)。（过滤震荡市中的无意义信号）。
  * 天地针熔断：计算该笔交易的止损距离百分比 (如 `(High - Low) / EntryPrice`)，必须 <= 系统配置的 `max_sl_dist` (如 3.5%)。超过则视为极端行情，直接抛弃。
* **点位计算**：
  * 入场价 (`entry_price`)：K线收盘价。
  * 止损价 (`stop_loss`)：该 Pinbar 的极值点（多单为最低点，空单为最高点）。
  * 止盈价 (`take_profit_1`)：严格按照 1.5R（1.5倍止损距离）计算。

### 3.2 资金与风控模型 (Position Sizing)
不使用固定杠杆，采用动态推算模型：
* **前置拦截**：当前持仓笔数 $n \ge 4$ 时，拒绝生成建议，拦截信号。
* **分配本金计算**：可用资金 $Investment = TotalWalletBalance / (4 - n)$。
* **固定风险金额**：单笔最大允许亏损额 = $TotalWalletBalance \times risk\_pct$ (默认 2%)。
* **名义价值计算**：理论开仓名义价值 = 风险金额 / 止损距离百分比。
* **动态杠杆**：计算理论杠杆 = 名义价值 / $Investment \times 1.05$ (预留维持保证金缓冲)。
* **杠杆熔断**：若理论杠杆 > 系统配置的 `max_leverage` (如 20x)，则强制截断为 20x，并等比例反向压缩开仓数量，绝对不允许越过安全红线。

## 4. 非功能性需求与架构约束
1. **领域驱动设计 (DDD)**：
   * `core/entities.py`：必须使用纯 Python `@dataclass`，没有任何第三方库依赖。
   * `domain`：纯数学与逻辑计算，没有任何 I/O 操作。
2. **异步与高并发**：
   * 所有网络请求 (httpx) 和数据库读写 (aiosqlite) 必须使用 `async/await` 非阻塞写法。
   * 消息推送必须使用 `asyncio.gather` 并发执行。
3. **动态配置中心**：
   * 提供基于 FastAPI 的 Web 接口 (`/api/config`)。
   * 参数必须通过 `Pydantic` 进行严格的上下界校验 (如 leverage 必须在 1~125 之间)。

## 5. 前端 UI/UX 规范 (Web Dashboard)
系统需提供一个单页面应用 (SPA) 作为配置控制台。
* **设计风格**：苹果简洁科技风 (Apple Minimalist & Tech Style)。
* **视觉要素**：
  * 支持深色模式 (Dark Mode)，背景采用深邃的科技黑或深灰。
  * 卡片采用毛玻璃效果 (Glassmorphism)，带有细腻的阴影和圆角 (Rounded-2xl)。
  * 字体优先使用系统级无衬线字体 (如 SF Pro, Inter, -apple-system)。
  * 交互动画需如丝般顺滑 (Smooth Transitions)，例如表单提交时的 Loading 态和成功提示 (Toast)。
* **技术栈**：纯 HTML + Tailwind CSS (通过 CDN 引入) + 原生 JavaScript (Fetch API)。不使用任何需要编译的重型框架 (如 React/Vue CLI)。
* **交互逻辑**：页面加载时 GET `/api/config` 渲染表单默认值；点击保存时 PUT `/api/config`，并包含 Pydantic 级别的基础表单前端校验。
