# 重构验收检查清单

## 验收流程

```
1. 前后端各自开发完成
   ↓
2. 代码审查（AI 审查，提出建议）
   ↓
3. 前后端整改
   ↓
4. 接口联调
   ↓
5. 端到端测试
   ↓
6. 验收通过
```

---

## 第一部分：代码审查（不直接修改，只提建议）

### 后端审查要点

#### DDD 合规性
- [ ] `domain/` 层是否包含任何 I/O 操作（async/await、httpx、aiosqlite）？
- [ ] `application/` 层是否直接访问数据库而非通过仓储接口？
- [ ] `infrastructure/` 层是否实现了 `core/interfaces.py` 中的抽象接口？
- [ ] 实体类是否仅包含数据，业务逻辑是否在领域服务中？

#### 代码质量
- [ ] 类型注解是否完整？
- [ ] 是否有未处理的异常？
- [ ] SQL 是否有参数化（防止注入）？
- [ ] 日志是否清晰可追踪？

#### 性能
- [ ] 分页查询是否有 `LIMIT/OFFSET`？
- [ ] 常用查询字段是否有索引？
- [ ] 是否有 N+1 查询问题？

#### API 设计
- [ ] 返回格式是否统一？
- [ ] 错误响应是否有明确信息？
- [ ] 默认参数是否合理？

---

### 前端审查要点

#### TypeScript 质量
- [ ] 所有组件是否有 Props 类型定义？
- [ ] API 响应是否有 Interface 定义？
- [ ] 是否有 `any` 类型滥用？

#### React 最佳实践
- [ ] 是否有不必要的状态？
- [ ] useEffect 依赖是否完整？
- [ ] 是否有内存泄漏（未清理的订阅）？

#### 用户体验
- [ ] 加载状态是否明确？
- [ ] 错误提示是否友好？
- [ ] 空状态是否有提示？

#### 性能
- [ ] 是否有不必要的重渲染？
- [ ] 列表是否有 key？
- [ ] 大量数据是否有虚拟滚动？

## 第四部分：端到端场景测试

### 场景 1：用户查看信号列表
1. 打开应用
2. 切换到"信号雷达"标签
3. 不设置条件，点击查询
4. 查看第一页信号
5. 翻页到第二页
6. 设置币种筛选为 BTCUSDT
7. 点击查询
8. 验证仅显示 BTCUSDT 信号

**验收点**: 页面响应流畅，数据正确，分页正常

### 场景 2：用户查看持仓
1. 切换到"用户持仓"标签
2. 查看当前持仓
3. 点击刷新按钮
4. 验证数据更新

**验收点**: 刷新按钮有效，数据实时更新

### 场景 3：用户查看仪表盘
1. 切换到"仪表盘"标签
2. 验证三个余额字段显示
3. 手动计算验证：保证金余额 = 钱包余额 + 未实现盈亏

**验收点**: 数据准确，计算正确

## 第五部分：改进建议检查表

### 高优先级（必须整改）
- [ ] 安全漏洞（SQL 注入、XSS）
- [ ] 数据不一致
- [ ] 功能缺陷
- [ ] 严重性能问题

### 中优先级（建议整改）
- [ ] 代码重复
- [ ] 类型不完整
- [ ] 错误处理不统一
- [ ] 用户体验不佳

### 低优先级（可选优化）
- [ ] 代码风格统一
- [ ] 注释完善
- [ ] 日志优化
- [ ] 视觉细节
