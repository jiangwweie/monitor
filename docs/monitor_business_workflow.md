# monitor 系统核心业务流程与指标展示逻辑全案

本文档整理了系统全栈升级与优化 4 落地后的最新业务流水线设计、功能模块划分及前端数据展现逻辑。

## 1. 系统核心定位 (Core Identity)
- **品牌名称**：monitor (加密货币信号监测系统)
- **安全底线约束 (Zero Execution)**：系统被严格限定为“纯信号监测与风控评估工具”。系统底层的自动化下单接口 (`auto_order_status`) 被强制硬编码在 `OFF` 状态，并且在配置中心前端界面上始终置灰锁定，从架构级别杜绝误操作导致真实资金损失的可能。

## 2. 前端模块与展示核心逻辑 (Frontend Architecture & Display Logic)

系统中台目前划分为三大核心视图模块，每一个视图都承担了明确的数据展现和交互职责：

### 2.1 仪表盘与大屏监控 (Dashboard)
负责全局遥测数据以及真实持仓状况的宏观呈现，必须保证 7x24 小时高可用。

- **顶层系统健康度 (System Health)**：
  - **WebSocket 脉冲心跳**：通过动态指示灯区分服务在线与离线边缘状态。
  - **API 限流防护伞**：实时展示当前调用交易所 API 积累的“权重消耗百分比”。配合独特的**每分钟重置倒计时**仪表，在水位超过 80% 时会自动变红预警，支撑系统能够在多币种高频并发下规避 IP 封禁惩罚。
- **盘口价格轮播 (Realtime Ticker)**：在卡片顶部核心渲染区，基于用户设置的活跃币种，实时并排展示多币对动态成交价格。
- **关键资产透视 (Account & Margin)**：剥离多余字段，聚焦展示交易员最关心的三大指标：**账户总资金**、**可用保证金**，以及根据活口汇兑得出的**总计未实现盈亏**。
- **分笔持仓详情反馈**：使用极具苹果属性的“毛玻璃卡片流”平铺活跃持仓。包含：开仓方向（红绿标识）、币别、使用杠杆、入场均价、持仓粒度，且最重要的是**实时闪烁的单笔浮动盈亏（Unrealized PnL）**。

### 2.2 信号雷达与检索中枢 (Signal Radar)
承载所有量化策略引擎（如 Pinbar+EMA60）扫描生成的只读快照池，注重信息密度与检索效率。

- **高密度列表展示**：表头仅暴露最直接的决策字段：发生时间、币种、方向、系统加权综合评分 (Score)、建议入场价。
- **多维度复合查询引擎**：
  - 用户可借助顶部的组合选择器，通过**交易对** (BTCUSDT 等)、**多空方向** (LONG/SHORT)、甚至**时间流逝窗口** (最近 1 小时 / 最近 24 小时) 迅速捞取所需信号。
- **深度穿透弹窗 (Deep Dive Dialog)**：
  - 由于列表空间有限，当用户点击“详情”时，触发侧边半透明面板。
  - **价格与风控区间**：展示第一止盈位 (TP1)、绝对止损位 (SL)、风控止损距离占比。
  - **微观打分与内部算力**：拆解呈现影响 `Score` 的核心原因 (Reason)、具体的形态分/趋势分/波动爆发分，并且露出被隐藏的硬核分析数据（影线真实占比、偏离 EMA 距离、ATR 波动率）。
- **空间管理**：赋予人工介入能力，支持利用勾选框完成多信号批量删除，保持视野整洁。

### 2.3 系统热配置中心 (Settings)
所有动态策略与消息路由的中央控制板，具备无感热重载能力。

- **通知分发网关 (Webhook Hub)**：拥有“全局总开关”掌控一切通知命脉；并行支持“飞书机器人”与“企业微信”双推交通路。填写的凭据 (Secret Key) 在界面均采取星号遮罩，并在后端闭环强制加密。
- **多币种无限拓扑**：对于扫描标的，除系统内化的预设四大主流资产之外，开放“+ 添加”探针，容许交易员随时跨交易所注入冷门或新兴代币加入队列监控。
- **恒定分配评分盘控制**：将抽象的打分逻辑 ($W_{shape} + W_{trend} + W_{vol}$) 具象化为相互联动的三个动态推拉阻尼条。推高任何一个指标的具体占比后，系统前端脚本将自动平滑削减其他指标的权重额度，确保总比重永远卡死在 100% 的数学临界点。

## 3. 后端数据流向与底层编排模型 (Backend Data Flow)
- **领域模型承载 (Entities & DDD)**：所有前端看到的计算数据全部被抽象为纯净的 Python `@dataclass` 结构，比如封装了信号矩阵的 `Signal` 对象，或者封装查询的 `SignalFilter` 与翻滚状态 `SystemStatus`，实现了展示需求与存取底层的完美解耦。
- **严苛的类型与安全屏障**：前端提交的任一改动命令 (通过 `PUT /api/config`)，必经后置的 Pydantic 校验池过滤，若不符合格式验证或跨越数字红线边界，则原路驳回拦截，保障运行时常青。
