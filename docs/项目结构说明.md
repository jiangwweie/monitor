# 项目结构说明

## 项目概述

本项目是一个加密货币信号监测系统，采用 Clean Architecture / DDD（领域驱动设计）架构，遵循 Zero Execution Policy（零执行策略），仅提供信号监测和风控评估功能，绝不执行实际交易。

## 目录结构

```
monitor/
├── core/               # 核心层：实体、接口、异常定义
├── domain/             # 领域层：业务逻辑、策略引擎、风控计算
├── application/        # 应用层：业务编排、服务调度
├── infrastructure/     # 基础设施层：外部适配器、持久化、通知
├── web/                # Web 层：FastAPI 路由、REST 接口
├── web_ui/             # 前端：React 19 + TypeScript SPA
├── scripts/            # 脚本：维护工具、测试脚本
├── tests/              # 测试：单元测试、集成测试
├── config/             # 配置：Docker、Nginx 等配置文件
├── docs/               # 文档：系统设计、API 契约、使用说明
├── logs/               # 日志：应用运行日志
├── main.py             # 入口：依赖注入、FastAPI 启动
├── requirements.txt    # Python 依赖
└── README.md           # 项目说明
```

## 各模块详细说明

### 1. core/ - 核心层

**职责**：定义系统最内层的抽象，包含领域实体、接口规范、异常定义。无外部依赖，仅使用 Python 标准库。

| 文件 | 功能说明 |
|------|----------|
| `entities.py` | 核心数据实体定义：`Bar`(K 线)、`Signal`(信号)、`Position`(持仓)、`AccountBalance`(账户余额)、`PositionSizing`(仓位计算结果)、`RiskConfig`(风控配置)、`ScoringWeights`(评分权重)、`PinbarConfig`(Pinbar 配置)、`SignalFilter`(信号过滤条件)、`SystemConfig`(系统配置)、`SystemStatus`(系统状态) 等 |
| `interfaces.py` | 外部接口抽象：`IDataFeed`(行情源)、`IAccountReader`(账户读取器)、`IExchangeAPI`(交易所 API)、`IRepository`(持久化仓储)、`IConfigProvider`(配置提供者)、`INotifier`(通知推送器)。所有接口均使用 ABC 抽象基类定义 |
| `exceptions.py` | 业务异常定义：`RiskLimitExceeded`(风控超限异常)，用于交易信号或账户状态超过风控阈值时抛出 |

---

### 2. domain/ - 领域层

**职责**：实现核心业务逻辑，包含策略引擎、风控计算、评分系统。纯数学计算，无网络请求或数据库操作。

#### domain/strategy/ - 策略模块

| 文件 | 功能说明 |
|------|----------|
| `pinbar.py` | **Pinbar 形态识别策略引擎**：核心策略实现，包含形态评估、趋势过滤、波动率分析、动态止损计算、信号质量分级（A/B/C三级）。使用 EMA60 趋势判断和 ATR 波动率过滤 |
| `indicators.py` | 技术指标计算：`calculate_ema`(指数移动平均)、`calculate_atr`(真实波幅)，纯 Python 实现，无第三方依赖 |
| `scoring.py` | 动态评分引擎：提供多模式打分接口，支持经典模式、累进模式、自定义模式，向后兼容旧版权重配置 |
| `scoring_config.py` | 评分配置实体：`ScoringConfig`数据类，包含经典模式和累进模式的所有可调参数 |
| `scoring_factory.py` | 评分策略工厂：`ScoringStrategyFactory` 根据配置模式返回对应的策略实例，支持动态注册新策略 |
| `scoring_strategy.py` | 评分策略实现：`IScoreStrategy` 接口及三种实现——`ClassicScoreStrategy`(经典线性打分)、`ProgressiveScoreStrategy`(累进加分)、`CustomScoreStrategy`(自定义公式) |

#### domain/risk/ - 风控模块

| 文件 | 功能说明 |
|------|----------|
| `sizer.py` | **仓位计算器**：根据账户余额、风控配置和信号止损距离，计算建议杠杆、开仓数量、风险金额。使用 `Decimal` 保证精度，包含杠杆熔断机制 |
| `portfolio_risk.py` | **投资组合风险聚合服务**：计算所有持仓的总风险敞口，监控组合风险比例，防止多仓位同时止损导致总风险失控 |

#### domain/services/ - 领域服务

| 文件 | 功能说明 |
|------|----------|
| `account_service.py` | 账户服务：账户相关领域逻辑（待扩展） |
| `signal_service.py` | 信号服务：信号相关领域逻辑（待扩展） |
| `config_service.py` | 配置服务：系统配置管理（待扩展） |

### 3. application/ - 应用层

**职责**：编排领域对象和基础设施，实现业务流程调度。

| 文件 | 功能说明 |
|------|----------|
| `monitor_engine.py` | **核心引擎**：`CryptoRadarEngine` 主调度器，负责：数据获取 → 策略计算 → 账户读取 → 风控算仓 → 持久化 → 通知广播。包含冷启动预热、MTF 趋势校验、信号质量分级推送逻辑 |
| `history_scanner.py` | **历史扫描引擎**：`HistoryScanner` 异步扫描历史 K 线，回放策略检测，支持 MTF 穿越式趋势校验，任务状态管理，完成后推送汇总通知 |
| `chart_service.py` | 图表服务：K 线图表数据聚合，为前端提供 OHLCV 数据 |
| `signal_query_service.py` | 信号查询服务：应用层信号查询逻辑，提供分页、过滤、排序功能 |
| `position_service.py` | 持仓服务：应用层持仓管理逻辑，对接账户读取器和仓储 |

### 4. infrastructure/ - 基础设施层

**职责**：实现外部系统适配器，将技术细节与业务逻辑隔离。

#### infrastructure/feed/ - 行情源

| 文件 | 功能说明 |
|------|----------|
| `binance_ws.py` | **Binance WebSocket 适配器**：`BinanceWSFeed` 实现 `IDataFeed` 接口，订阅多币种多级别 K 线流，自动重连，只处理已闭合 K 线 (`is_closed=True`) |
| `binance_kline_fetcher.py` | K 线历史拉取器：`BinanceKlineFetcher` 通过 REST API 批量获取历史 K 线，支持进度回调 |

#### infrastructure/reader/ - 账户读取器

| 文件 | 功能说明 |
|------|----------|
| `binance_api.py` | **Binance HTTP 账户适配器**：`BinanceAccountReader` 实现 `IAccountReader` 接口，获取账户余额、持仓详情、止盈止损挂单。HMAC SHA256 签名，自动时间同步，**纯只读** |

#### infrastructure/repo/ - 持久化仓储

#### infrastructure/notify/ - 通知推送

| 文件 | 功能说明 |
|------|----------|
| `broadcaster.py` | **通知广播器**：`NotificationBroadcaster` 使用组合模式，实现多渠道并发推送，单一渠道故障不影响其他渠道 |
| `feishu.py` | 飞书机器人适配器：发送 Markdown 格式推送 |
| `wecom.py` | 企业微信适配器：发送 Markdown 格式推送 |
| `telegram.py` | Telegram Bot 适配器（已停用） |

#### infrastructure/utils/ - 工具类

| 文件 | 功能说明 |
|------|----------|

### 5. web/ - Web API 层

| 文件 | 功能说明 |
|------|----------|
| `api.py` | **FastAPI 路由**：提供 RESTful API 接口，包括：系统状态、信号查询/删除、配置热更新、历史扫描任务、图表数据、持仓查询、配置导入导出等 |

---

### 6. main.py - 系统入口

**职责**：依赖注入编排、FastAPI 生命周期管理、引擎启动。

**关键流程**：
1. 加载环境变量并验证
2. 实例化领域对象（`PinbarStrategy`、`PositionSizer`）
3. 实例化基础设施（`BinanceWSFeed`、`BinanceAccountReader`、`SQLiteRepo`、通知推送器）
4. 组装 `CryptoRadarEngine` 并挂载到 `app.state`
5. 启动后台任务：`asyncio.create_task(engine.start())`
6. 生命周期结束时平滑关闭引擎

## 关键设计模式

### 1. 依赖注入 (Dependency Injection)
所有组件在 `main.py` 中手动实例化并传递，无全局单例，便于测试和替换。

### 2. 接口隔离 (Interface Segregation)
外部依赖通过 `core/interfaces.py` 中的抽象接口访问，便于 Mock 测试和切换实现。

### 3. 组合模式 (Composite Pattern)
`NotificationBroadcaster` 将多个通知渠道组合为单一对象，统一调用。

### 4. 策略模式 (Strategy Pattern)
`ScoringStrategyFactory` 根据配置返回不同的评分策略实现。

### 5. 工厂模式 (Factory Pattern)
评分策略、信号分级等使用工厂方法创建实例。

## 核心设计理念

1. **Zero Execution Policy**：`auto_order_status` 硬编码为 `OFF`，系统仅为信号观测工具
2. **Clean Architecture**：依赖方向始终向内（infrastructure → application → domain → core）
3. **Zero Side Effects in Domain**：领域层纯函数，无网络/数据库操作
4. **Defensive Programming**：引擎外层 try-except 保证 7x24 小时运行
5. **并发隔离**：`asyncio.gather(return_exceptions=True)` 防止单一渠道故障雪崩
