# 🚀 CryptoRadar 纯信号监测与风控系统 - 核心系统需求与架构设计说明书

## 一、 项目背景与系统定位

**CryptoRadar** 是一款专注于加密货币合约市场的“纯信号监测与决策辅助系统”。

- **核心目标**：7x24 小时监听实时行情，基于定制的量化策略（Pinbar + EMA60）寻找高胜率交易机会，结合当前账户真实余额，计算出严格遵守风控规则的**建议开仓量与动态杠杆**，最终通过多渠道（飞书/Telegram/企业微信）发送富文本决策卡片。
- **安全底线（绝对约束）**：本系统属于**只读雷达（Read-Only Radar）**。系统架构内**绝对禁止**包含任何向交易所发送真实下单（如 `POST /fapi/v1/order`）的执行代码。它仅读取行情和账户余额。

## 二、 技术栈与架构范式

- **编程语言**：Python 3.10+
- **核心框架**：`asyncio` (全异步非阻塞核心), `httpx` (异步网络请求)
- **Web 与配置层**：`FastAPI`, `pydantic` v2, `uvicorn`
- **持久化层**：`aiosqlite` (异步 SQLite)
- **架构范式**：严格遵循 **领域驱动设计 (DDD)** 和 **六边形架构 (端口与适配器)**。
- **设计模式要求**：依赖注入 (Dependency Injection)、组合模式 (Composite Pattern 用于多渠道推送)、策略模式 (Strategy Pattern)。

## 四、 核心数据模型与契约接口 (`core/`)

AI 在编写业务逻辑前，必须先实现以下底层契约。此处没有任何第三方库依赖。

### 1. 核心实体 (`core/entities.py`)

- `Bar`: 包含 `symbol`, `interval`, `timestamp`, `open`, `high`, `low`, `close`, `volume`, `is_closed`。
- `Signal`: 包含 `symbol`, `direction` (LONG/SHORT), `entry_price`, `stop_loss`, `take_profit_1` (基于 1.5R 盈亏比), `score`, `indicators_snapshot` (字典，存放当时的 EMA/ADX 值)。
- `AccountInfo`: 包含 `total_wallet_balance` (钱包总额 USDT), `available_balance` (可用余额), `position_count` (当前真实持仓笔数，用于 4-n 逻辑)。
- `SizingRecommendation`: 包含 `qty` (建议开仓数量), `leverage` (建议杠杆倍数), `notional` (名义价值), `actual_risk_pct` (实际承担风险比例)。

### 2. 抽象接口 (`core/interfaces.py`)

- `IDataFeed`: 定义异步生成器 `stream_closed_bars(symbol, interval) -> AsyncGenerator[Bar, None]`。
- `IAccountReader`: 定义 `get_account_info() -> AccountInfo`。
- `INotifier`: 定义 `send_markdown(title: str, content: str) -> bool`。
- `IRepository`: 定义 `save_signal(...)`, `get_app_config()`, `update_app_config(...)`。

## 五、 核心领域逻辑约束 (`domain/`)

### 1. 策略大脑 (`domain/strategy/pinbar.py`)

- **输入**：最新闭合的 `Bar`，历史 `Bar` 序列（用于计算指标），动态配置 `SystemConfig`。
- **形态规则**：实体长度 $\le$ 全长的 25%，影线长度 $\ge$ 实体的 2.5 倍。波幅（最高点-最低点）必须 $> 1.2 * ATR(14)$，且波幅不得超过当前价格的 3.5%（过滤天地针）。
- **趋势过滤**：收盘价在 EMA60 之上，仅寻找看涨 Pinbar；反之寻找看跌 Pinbar。
- **输出**：满足所有条件返回 `Signal` 对象，否则返回 `None`。止损位(`stop_loss`)设为 Pinbar 极值外加极小缓冲。止盈位(`take_profit_1`)必须基于 1.5 倍止损距离 (1.5R) 动态计算。

### 2. 风控与头寸计算 (`domain/risk/sizer.py`)

- **输入**：`Signal`, `AccountInfo`, `RiskConfig` (来自数据库热加载)。
- **逻辑 1 (头寸可用性)**：读取 `AccountInfo.position_count`，若 $\ge 4$，直接抛出 `RiskLimitExceeded` 异常拦截。
- **逻辑 2 (基础资金)**：分配可用资金 $Investment$ = `AccountInfo.total_wallet_balance / (4 - position_count)`。（若 position_count 为 0，则直接取 available_balance / 4）。
- **逻辑 3 (固定风险模型)**：计算止损距离百分比 $SL_{dist}$。单笔风险金额 = 总资金 $\times$ `RiskConfig.risk_pct`。理论名义价值 = 风险金额 / $SL_{dist}$。
- **逻辑 4 (动态杠杆)**：计算理论杠杆 = 名义价值 / $Investment$ $\times 1.05$。若超过 `RiskConfig.max_leverage`，则熔断降级至 `max_leverage`，并反向压缩数量。
- **输出**：返回严格截断价格精度后的 `SizingRecommendation`。

## 六、 基础设施层实现要求 (`infrastructure/`)

- **数据库 (`sqlite_repo.py`)**：使用 `aiosqlite`。必须支持配置表 `app_configs` 的热更新 (`ON CONFLICT DO UPDATE`)。必须异步非阻塞。
- **多渠道并发推送 (`broadcaster.py`)**：实现 `INotifier` 接口，内部持有一个 `List[INotifier]`。`send_markdown` 方法必须使用 `asyncio.gather(*tasks, return_exceptions=True)` 并发调用所有子渠道，确保单个渠道网络超时不会导致系统雪崩。

------

## 七、 调度引擎流水线 (`application/monitor_engine.py`)

实现 `SignalMonitorEngine` 类，包含核心循环逻辑：

1. 接收底层的 K 线流。
2. 每接收到一根闭合 K 线，首先从 DB 异步拉取最新动态配置 (`RiskConfig`, `SystemConfig`)。
3. 调用 `PinbarStrategy`，若返回 `None` 则跳过。
4. 调用 `IAccountReader` 获取真实余额。
5. 调用 `PositionSizer` 计算开仓建议。捕获 `RiskLimitExceeded` 异常并仅记录日志。
6. 将 `Signal` 和 `SizingRecommendation` 异步存入数据库。
7. 调用 `NotificationBroadcaster.send_markdown()` 发送富文本决策卡片。卡片内容需清晰展示入场价、止盈止损价、算出的杠杆倍数和数量。

## 八、 Web 参数热更新层 (`web/api.py`)

- 使用 FastAPI 编写独立的 REST API。
- 提供 `GET /api/config` 读取 SQLite 中 `app_configs` 表的内容。
- 提供 `PUT /api/config` 接收 Pydantic Payload。必须利用 Pydantic 的 `Field(ge=x, le=y)` 进行极度严格的安全校验。
- 写入 SQLite 后返回成功响应。底层的 `monitor_engine.py` 会在下一次处理 K 线时自动读到新配置，实现热加载。
